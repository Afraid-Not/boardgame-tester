# 보드게임 밸런스 테스터 — 프로젝트 기획서

## 1. 프로젝트 개요

### 1.1 목표

보드게임 규칙서를 입력하면 AI가 자동으로 게임 환경을 생성하고, 강화학습(RL) 기반 시뮬레이션을 통해 밸런스를 분석하며, 구체적인 개선안을 제시하는 B2B SaaS 플랫폼.

### 1.2 타겟 유저

- 보드게임 회사 (퍼블리셔, 디자이너)

### 1.3 지원 게임 장르

- **경제 기반 보드게임** (부루마블, 모노폴리 류) — 자원 관리, 주사위, 맵 기반
- **카드 대전 게임** (하스스톤, 유희왕 류) — 카드 스탯, 덱 구성, 턴 기반

### 1.4 핵심 기능

| 기능            | 설명                                                           |
| --------------- | -------------------------------------------------------------- |
| 규칙 파싱       | 규칙서 텍스트/이미지 업로드 → AI가 게임 구조 파싱              |
| 게임 환경 생성  | 파싱된 규칙 기반으로 RL 학습 가능한 코드 환경 자동 생성        |
| RL 시뮬레이션   | 에이전트끼리 수천~수만 판 자동 플레이                          |
| 밸런스 분석     | 선/후공 승률, 전략별 승률, 밸런스 비율 리포트                  |
| 개선 가이드라인 | 불균형 감지 시 구체적 수치 조정 제안 (예: "카드 A 공격력 5→3") |
| 재검증          | 가이드라인 적용 후 재시뮬레이션으로 개선 확인                  |

## 2. 시스템 아키텍처

```
[사용자] → [Next.js 프론트엔드]
               ↓
         [FastAPI 백엔드]
               ↓
    ┌──────────┼──────────┐
    ↓          ↓          ↓
[Claude API] [벡터 DB]  [Celery + Redis]
 규칙 파싱    RAG 참조    작업 큐 관리
    ↓                      ↓
[게임 환경 코드 생성]   [GPU Worker 1~7]
                        RL 학습 실행
                           ↓
                     [결과 분석 Agent]
                           ↓
                    [밸런스 리포트 + 가이드라인]
```

### 2.1 인프라 구조

```
[클라우드]                        [로컬 GPU 클러스터]
 ├── Vercel → Next.js Frontend    ├── Worker 1: RTX 3050
 ├── Supabase → DB + Auth         ├── Worker 2: RTX 3050
 │   + Storage + pgvector         ├── Worker 3: RTX 3050
 └── GitHub → 코드 관리           ├── Worker 4: RTX 3050
                                  ├── Worker 5: RTX 3050
[로컬 서버]                       ├── Worker 6: RTX 3050
 ├── FastAPI (ngrok으로 외부 노출) └── Worker 7: RTX 5070 Ti
 └── Redis (Message Broker)
```

- Frontend는 Vercel에 배포, Backend는 로컬 FastAPI + ngrok
- DB/Auth/Storage는 Supabase 사용 (pgvector 포함)
- RL 학습은 로컬 GPU 머신에서 Celery Worker로 실행
- 각 머신이 서로 다른 게임/설정을 병렬로 학습
- 코드 관리는 GitHub

## 6. 주요 데이터 모델

### 6.1 Game (게임 정보)

```
Game
├── id
├── name (게임 이름)
├── genre (card_battle | economic_board)
├── rules_text (파싱된 규칙 텍스트)
├── rules_raw (원본 규칙서 파일 경로)
├── parsed_structure (JSON - 파싱된 게임 구조)
├── environment_code (생성된 RL 환경 코드 경로)
├── status (parsing | ready | training | completed)
├── created_at
└── updated_at
```

### 6.2 TrainingJob (학습 작업)

```
TrainingJob
├── id
├── game_id (FK → Game)
├── algorithm (PPO | DQN | A2C 등)
├── hyperparameters (JSON)
├── worker_id (할당된 GPU Worker)
├── status (queued | running | completed | failed)
├── progress (0~100%)
├── total_episodes
├── started_at
└── completed_at
```

### 6.3 BalanceReport (밸런스 리포트)

```
BalanceReport
├── id
├── training_job_id (FK → TrainingJob)
├── win_rates (JSON - 선/후공별, 전략별 승률)
├── balance_score (0~100, 50이 완벽한 밸런스)
├── dominant_strategies (JSON - 지배 전략 목록)
├── severity (good | warning | critical)
├── guidelines (JSON - 개선 제안 목록)
├── created_at
```

### 6.4 GameReference (기존 게임 데이터 - RAG용)

```
GameReference
├── id
├── name (게임 이름)
├── genre
├── rules_summary
├── balance_data (JSON - 알려진 밸런스 정보)
├── strategies (JSON - 유효 전략 목록)
├── embedding (벡터 - pgvector)
└── source (데이터 출처)
```

## 8. 핵심 파이프라인 상세

### 8.1 규칙 파싱 파이프라인

```
[규칙서 입력 (텍스트/이미지)]
    ↓
[Claude API - Vision + Text]
    ↓
[RAG: 벡터 DB에서 유사 게임 검색]
    ↓
[유사 게임 참조하여 구조화된 게임 JSON 생성]
    ↓
[게임 JSON → RL 환경 코드 자동 생성]
    ↓
[코드 검증 (자동 테스트)]
    ↓
[저장]
```

**게임 JSON 구조 예시 (카드게임):**

```json
{
  "type": "card_battle",
  "players": 2,
  "turn_based": true,
  "components": {
    "cards": [
      { "name": "전사", "attack": 5, "defense": 3, "cost": 3 },
      { "name": "마법사", "attack": 7, "defense": 1, "cost": 5 }
    ],
    "deck_size": 30,
    "hand_size": 5
  },
  "win_condition": "opponent_hp_zero",
  "phases": ["draw", "play", "attack", "end"]
}
```

### 8.2 RL 학습 파이프라인

```
[게임 환경 코드 로드]
    ↓
[Celery Task 생성 → Redis Queue]
    ↓
[GPU Worker가 Task 수신]
    ↓
[Stable-Baselines3 학습 실행]
  ├── Self-play (에이전트 vs 에이전트)
  ├── 전략 탐색 (다양한 정책 학습)
  └── 선/후공 교대 학습
    ↓
[학습 결과 저장 (모델 + 로그)]
    ↓
[결과 분석 Agent로 전달]
```

### 8.3 밸런스 분석 파이프라인

```
[학습된 모델로 평가 게임 N판 실행]
    ↓
[통계 수집]
  ├── 선/후공 승률
  ├── 전략별 승률 분포
  ├── 게임 길이 분포
  └── 카드/유닛별 사용률 + 승률 기여도
    ↓
[밸런스 점수 산출 (0~100)]
    ↓
[불균형 감지 시 → Claude API로 개선 가이드라인 생성]
  ├── 구체적 수치 조정 제안
  ├── 규칙 변경 제안
  └── 예상 개선 효과
    ↓
[가이드라인 적용 → 게임 환경 수정 → 재학습 → 재분석]
    ↓
[Before/After 비교 리포트 생성]
```

## 10. 리스크 및 대응

| 리스크                     | 영향                  | 대응                                                                       |
| -------------------------- | --------------------- | -------------------------------------------------------------------------- |
| 규칙 파싱 정확도 부족      | 잘못된 게임 환경 생성 | 사용자가 파싱 결과 검토/수정 가능한 UI 제공, RAG로 품질 보완               |
